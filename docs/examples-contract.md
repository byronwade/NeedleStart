# Examples And Templates Contract

Status: Planned.

Audience: app developers, framework contributors, docs maintainers, AI agents.

This page defines how Lumina examples, starter templates, and public guide snippets should work once implementation exists. The repository is in Phase 1 scaffold, so no example is fully verified yet. Scaffolded source fixtures exist for `apps/www`, `examples/basic/`, `examples/blog-seo/`, `examples/multi-app-workspace/`, `examples/large-100-routes/`, and `examples/large-1000-routes/`; `apps/www` is runnable through the minimal local dev-server path and static build/start path, while standalone examples remain scaffolded until their own command evidence, generated artifacts, and runtime limitations are documented. The planned official inventory lives in [Examples Catalog](examples-catalog.md).

## Research Notes

Mature framework docs make examples part of the product surface, not an afterthought:

- [Next.js `create-next-app`](https://nextjs.org/docs/app/api-reference/cli/create-next-app) supports default app creation and creation from examples.
- [Next.js project structure](https://nextjs.org/docs/app/getting-started/project-structure) documents folder and file conventions close to onboarding.
- [Astro getting started](https://docs.astro.build/en/getting-started/) links project creation, starter templates, and a tutorial path.
- [Astro recipe writing guidance](https://contribute.docs.astro.build/guides/recipe-writing/) treats recipes as practical, outcome-focused walkthroughs.
- [Vite getting started](https://vite.dev/guide/) keeps scaffolding commands concise and template-driven.
- [Vite SSR guide](https://vite.dev/guide/ssr) points to example projects for advanced rendering setups.

Lumina should learn from those patterns while staying honest about the current scaffold. Examples must be runnable evidence before they become marketing or onboarding claims.

## Example Categories

Planned example categories:

| Category | Purpose | Initial examples |
| --- | --- | --- |
| Starter template | Minimal app generated by the create command. | `basic` |
| Tutorial app | Step-by-step learning path that introduces core concepts. | `blog-seo` |
| Feature example | Focused example for one capability. | `api-route`, `hot-api`, `static-export`, `adapter-node` |
| Integration example | Shows a common external library or deployment pattern. | Planned after core behavior exists. |
| Agent example | Proves map inspection, safe edits, and agent context. | `agent-demo` |
| Scale fixture | Proves large-app discovery, graph output, and performance gates. | `playgrounds/large-app-fixture` |

Do not add integration examples before the relevant core contract is implemented and tested.

## Status Labels

Use these labels consistently:

| Status | Meaning | Public docs behavior |
| --- | --- | --- |
| Planned | Target example only. Files may not exist. | May be described as planned, not usable. |
| Scaffolded | Files exist, but core behavior is incomplete or unverified. | May be linked from contributor docs only. |
| Runnable | Commands start without known local failures, but full evidence is incomplete. | May be used in internal implementation notes. |
| Verified | Commands, tests, generated artifacts, and docs claims pass. | May be linked from public onboarding and guides. |
| Deprecated | Example remains for migration or compatibility notes. | Must link to replacement and removal timeline. |

An example cannot be called verified until the evidence requirements below are met in the current repository state.

## Minimum Example README

Every example README must include:

- Status label.
- Lumina version or workspace commit it targets.
- What the example demonstrates.
- Prerequisites.
- Commands to install, run, build, inspect, and test.
- Expected route list.
- Expected generated artifacts.
- Expected public HTML or endpoint behavior.
- Which docs contracts the example exercises.
- Known limitations.
- Troubleshooting section.
- Agent workflow notes when relevant.

Example READMEs should be short enough for a new user to follow, but exact enough for an agent to verify.

## Required Commands

Each verified example should document the smallest complete command set. `bun run dev`, `bun run build`, and `bun run start` are generated app package scripts; they must call the equivalent `lumina dev`, `lumina build`, and `lumina start` framework commands instead of becoming separate framework behavior.

Use generated app package scripts for the normal run/build/start path. Use direct `lumina ... --json` commands only when the example needs machine-readable framework inspection output that package scripts do not wrap.

```bash
bun install
bun run dev
bun run build
bun run start
bun test
lumina routes --json
lumina inspect --json
lumina check --json
```

Do not publish these as working commands until CLI behavior, example files, and verification evidence exist.

## Verification Evidence

An example is verified only when review evidence includes:

- Commands run from a clean checkout.
- Runtime versions.
- Exit codes.
- Generated artifact paths.
- Stable JSON or snapshot output.
- Public HTML or HTTP assertion when relevant.
- SEO assertion when the example includes public pages.
- Adapter output assertion when the example targets deployment.
- Performance or benchmark data only if the example makes a speed claim.

If an example cannot be verified on Windows, macOS, and Linux, document the known platform limitation in [Compatibility](compatibility.md).

## Create Command Integration

The future create command may support examples:

```bash
bun create lumina my-app
bun create lumina my-blog --example blog-seo
bun create lumina my-api --example api-route
```

The default generated app must be the smallest useful verified starter. Other examples can be selectable only after they meet the verified requirements.

The create command must not fetch arbitrary remote code silently. Remote templates, if supported later, need explicit source display, integrity policy, and security review.

## Public Docs Rules

Public docs may link a guide to an example only when:

- The example exists in the repository.
- The example README is current.
- The guide names the example status.
- The guide links to the relevant reference docs.
- The guide does not claim behavior works unless verification evidence exists.

Before that point, public guides may describe target workflows but must say they are planned.

## Agent Demo Requirements

The `agent-demo` example must eventually prove:

- Route context inspection.
- Lumina Map affected query.
- Safe metadata edit dry run.
- Safe metadata edit apply.
- Affected checks.
- Mutation log.
- Undo.
- Human sign-off behavior for high-risk edits.

The demo must not require production secrets, external network access, or hidden state.

## Scale Fixture Requirements

The `large-app-fixture` example must eventually prove:

- Deterministic route sorting across operating systems.
- Route discovery performance at a documented app size.
- Graph generation performance at a documented app size.
- Agent context size limits.
- Stable diagnostics for collisions and unsupported patterns.
- Public HTML and SEO checks for representative pages.

Scale fixture claims must link to [Benchmark Methodology](benchmark-methodology.md) and raw evidence.

## Out Of Scope For The Current Scaffold

- Runnable or verified example apps.
- Verified create-command output.
- Package installation commands.
- Remote template fetching.
- Public claims that examples run.
- Performance claims from example apps.

## Build Readiness

Before Lumina can call onboarding build-ready, it should have:

- A default starter example plan.
- A verified-example definition.
- A create-command relationship to examples.
- Example README requirements.
- Fixture and generated-artifact expectations.
- Public docs rules for linking examples.

This page provides that contract. Implementation must replace planned language with verified evidence as examples become real.
