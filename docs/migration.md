# Migration Tooling

Status: Planned.
Audience: app developers, migration contributors, AI agents.

Lumina should win adoption through the agent and map workflow, but migration friction still matters.

The migration story should start early as a prototype and mature after the core route, metadata, layout, and graph systems exist.

## Goals

- Preserve as much user code as possible.
- Generate route and component contracts when semantics are ambiguous.
- Make migration output inspectable through Lumina Map.
- Give agents useful context after migration.

## Initial Command

```bash
lumina migrate from-next
```

## Initial Scope

The first migration tool should handle a constrained subset:

- Next.js App Router pages.
- Layouts.
- Static metadata exports.
- Basic dynamic route segments.
- Public route metadata.
- Simple API route shape when possible.

## Contract Stub Generation

When semantics are ambiguous, the migration tool should generate `.contract.ts` stubs instead of guessing.

Example:

```ts
export default componentContract({
  name: "ProductCard",
  props: "unknown",
  confidence: 0.2,
  notes: [
    "Generated by migration. Review props and SEO role before relying on this contract."
  ],
})
```

## Migration Report

The command should emit:

- Files migrated.
- Files skipped.
- Contracts generated.
- Metadata preserved.
- Routes requiring manual review.
- Suggested affected checks.

## Agent Integration

After migration, agents should be able to run:

```bash
lumina map --json
lumina agent context --route /pricing --json
lumina check
```

## Out of Scope Initially

- Full Next.js parity.
- RSC migration.
- Middleware/proxy migration.
- Complex server action migration.
- Automatic auth migration.
- Automatic cache semantic migration.
