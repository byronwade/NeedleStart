# CLI Contract

The NeedleStart CLI is planned. This document defines the target command surface, output rules, and implementation contracts. Do not treat any command as implemented until `docs/status.md` marks it implemented or verified.

## Canonical Invocation

NeedleStart docs use these conventions:

| Context | Canonical command |
| --- | --- |
| Create a new app | `bun create needle my-app` |
| Run framework commands inside an app | `needle <command>` |
| Run local package scripts generated by the template | `bun run dev`, `bun run build`, `bun run start` |
| Work inside the NeedleStart monorepo | `bun install`, `bun test`, `bun run typecheck` |

Generated app scripts should call the CLI internally:

```json
{
  "scripts": {
    "dev": "needle dev",
    "build": "needle build",
    "start": "needle start",
    "check": "needle check"
  }
}
```

Documentation should prefer `needle dev`, `needle build`, and `needle start` when explaining the framework, and `bun run dev`, `bun run build`, and `bun run start` when explaining generated app scripts.

Avoid `bun needle <command>` in docs unless the project intentionally adds that runner form later.

## Design Goals

- Keep human output readable.
- Keep JSON output compact, deterministic, and schema-versioned.
- Make every command safe to call from agents.
- Make failures actionable with diagnostic codes.
- Avoid hidden network calls.
- Avoid using human-formatted output as the only source of truth.
- Prefer affected checks over whole-project work when graph data is available.
- Explain why the framework made render, cache, route, and graph decisions.

## Global Flags

All commands should eventually support:

| Flag | Meaning |
| --- | --- |
| `--json` | Emit stable machine-readable JSON. |
| `--pretty` | Pretty-print JSON for humans. |
| `--config <path>` | Use a specific config file. |
| `--root <path>` | Use a specific project root. |
| `--cwd <path>` | Execute as though started from another directory. |
| `--quiet` | Suppress non-error human output. |
| `--verbose` | Include extra human diagnostics. |
| `--ci` | Disable prompts and fail instead of asking. |
| `--no-color` | Disable ANSI color. |

Commands that can write files must also support:

| Flag | Meaning |
| --- | --- |
| `--dry-run` | Show the planned result without writing files. |
| `--force` | Allow explicitly gated high-risk behavior. |

`--force` must never bypass validation entirely. It only satisfies a documented human override requirement.

## Exit Codes

| Code | Meaning |
| --- | --- |
| `0` | Success. |
| `1` | General failure. |
| `2` | Invalid CLI usage or invalid flags. |
| `3` | Config load or validation failure. |
| `4` | Build, typecheck, render, route, graph, or manifest failure. |
| `5` | Check failure, such as SEO, budget, or affected tests. |
| `6` | Unsafe edit rejected. |
| `7` | Adapter capability mismatch. |
| `8` | MCP startup or protocol failure. |

JSON output should include the exit code in `exitCode` when the command is run with `--json`.

## JSON Output Envelope

Every `--json` command should use this envelope unless a command-specific contract says otherwise:

```ts
export type CliJsonEnvelope<T> = {
  schemaVersion: string
  command: string
  status: "pass" | "warn" | "fail"
  exitCode: number
  diagnostics: NeedleDiagnostic[]
  data: T
}
```

Rules:

- Use normalized POSIX-style paths.
- Sort arrays by stable keys.
- Avoid absolute paths unless explicitly requested.
- Include diagnostic codes for errors and warnings.
- Keep output stable enough for snapshot tests.
- Do not include secrets or raw environment values.
- Include `why` arrays for explain commands and framework decisions when available.

## Command Groups

### Create App

```bash
bun create needle my-app
bun create needle my-app --template basic
bun create needle my-app --template blog-seo
bun create needle my-app --template agent-demo
```

Owner: `create-needle`.

Responsibilities:

- Create a project directory.
- Install or prepare dependencies according to template policy.
- Write package scripts.
- Write starter `app/` files.
- Write `needle.config.ts`.
- Write starter `AGENTS.md` or generate it on first build.
- Clearly distinguish generated sample code from framework internals.

Out of scope initially:

- Interactive template marketplace.
- Remote template downloads.
- Auth, database, or CMS setup.

### Dev

```bash
needle dev
needle dev --host 0.0.0.0
needle dev --port 3000
needle dev --json
```

Owner: `@needle/cli`, `@needle/vite-plugin`, `@needle/compiler`.

Responsibilities:

- Load config.
- Discover routes.
- Start Vite middleware.
- Render React pages on the server.
- Hydrate client components.
- Watch route files.
- Update route manifest through HMR.
- Keep graph and SEO diagnostics current enough for local development.

Human output should include:

- Local URL.
- Selected adapter/runtime for dev.
- Route count.
- Warnings for invalid config or route collisions.
- Devtools URL when devtools exists.

### Build

```bash
needle build
needle build --json
needle build --adapter bun
needle build --adapter node
needle build --adapter static
```

Owner: `@needle/cli`, `@needle/compiler`, adapters.

Responsibilities:

- Load config.
- Discover routes.
- Classify render modes.
- Build route, render, SEO, cache, map, adapter, and perf manifests.
- Generate runtime modules.
- Run Vite build.
- Emit adapter-aware output.
- Run required checks unless disabled by explicit policy.

`needle build --json` should return a build summary with manifest paths, generated files, diagnostics, and adapter capabilities.

### Start

```bash
needle start
needle start --port 3000
needle start --host 0.0.0.0
needle start --json
```

Owner: selected adapter.

Responsibilities:

- Load built output.
- Start the selected runtime server.
- Serve static assets.
- Serve prerendered HTML.
- Route SSR, API, and hot API handlers.
- Apply cache headers.
- Preserve stable production error behavior.

`needle start` must not rediscover source routes.

### Routes

```bash
needle routes
needle routes --json
needle routes --route /pricing
```

Owner: `@needle/cli`, `@needle/compiler`.

Human output draft:

```txt
Route                  Kind   Mode       Source
/                      page   static     app/page.tsx
/blog/:slug            page   prerender  app/blog/[slug]/page.tsx
/api/health            api    api        app/api/health.ts
```

JSON `data` should include route IDs, paths, source files, kind, params, layouts, render mode, metadata status, and diagnostics.

### Inspect

```bash
needle inspect route /pricing
needle inspect file app/pricing/page.tsx
needle inspect node route:/pricing
needle inspect why route /pricing
needle inspect why file app/pricing/page.tsx
needle inspect route /pricing --json
```

Responsibilities:

- Explain what the compiler knows about a route, file, or graph node.
- Show source file, route mode, layouts, metadata, cache plan, related files, generated files, and recommended checks.
- Show why a route received its render mode, cache plan, SEO status, and graph relationships when that data is available.
- Avoid making semantic claims without graph confidence and `why` fields.

`needle inspect why` is an early wedge command. It should attack framework magic directly by answering why NeedleStart made a decision instead of only showing the final result.

Example JSON shape:

```json
{
  "route": "/pricing",
  "renderMode": {
    "mode": "static",
    "why": [
      "Route declares staticPage().",
      "No request-time APIs were detected.",
      "Metadata is statically analyzable."
    ]
  },
  "cache": {
    "mode": "static",
    "why": ["Static HTML is emitted at build time."]
  }
}
```

### SEO

```bash
needle seo
needle seo --route /pricing
needle seo --json
needle seo --fail-on warn
```

Owner: `@needle/seo`.

Responsibilities:

- Validate title, description, canonical URL, indexable HTML, Open Graph, structured data, sitemap inclusion, robots behavior, and relevant status codes.
- Emit route-level diagnostics.
- Provide safe fix suggestions when possible.

### Map

```bash
needle map
needle map --json
needle map route /pricing
needle map file components/ProductCard.tsx
needle map affected components/ProductCard.tsx
needle map explain edge:<edgeId>
```

Owner: `@needle/map`.

Responsibilities:

- Query the same graph engine used by MCP, Agent Kernel, and devtools.
- Show dependency edges with `kind`, `source`, `confidence`, and `why`.
- Recommend affected checks.
- Avoid hiding low-confidence inference.

### Agent

```bash
needle agent init
needle agent context --route /pricing --json
needle agent task "Update pricing hero copy"
needle agent plan "Update pricing hero copy" --json
needle agent log
```

Owner: `@needle/agent`.

Responsibilities:

- Generate or refresh agent-facing project context.
- Emit route context capsules.
- Produce deterministic task skeletons from templates and graph data.
- Read mutation logs.
- Avoid calling an external LLM as part of deterministic planning.

### MCP

```bash
needle mcp
needle mcp --read-only
needle mcp --stdio
needle mcp --http --port 3434
```

Owner: `@needle/mcp`.

Initial mode should be read-only.

Responsibilities:

- Expose route, map, SEO, cache, perf, and agent context resources.
- Keep output compact and deterministic.
- Avoid exposing secrets.
- Route all writes through the safe edit transaction engine once write tools exist.

### Edit

```bash
needle edit route /pricing --set meta.title="Pricing | Acme" --dry-run
needle edit route /pricing --set meta.title="Pricing | Acme"
needle edit transaction <mutationId>
needle edit undo <mutationId>
```

Owner: `@needle/agent` plus CLI wrapper.

Responsibilities:

- Validate target exists.
- Validate editable field.
- Assign risk tier.
- Apply AST-based patch in memory.
- Run affected checks.
- Return `SafeEditTransaction`.
- Append mutation log on apply.
- Support undo.

String replacement is not allowed for safe edits.

### Check

```bash
needle check
needle check --affected
needle check --since main
needle check --json
```

Owner: `@needle/cli` coordinating compiler, map, SEO, typecheck, tests, render, and budgets.

Responsibilities:

- Run the smallest reliable set of checks for a change.
- Use Needle Map when available.
- Fall back to broader checks when graph data is missing or low confidence.
- Return deterministic diagnostics for agents.

### Migrate

```bash
needle migrate from-next
needle migrate from-next --dry-run
needle migrate from-next --source ../next-app --target .
needle migrate from-next --json
```

Responsibilities:

- Convert compatible Next.js App Router pages.
- Preserve layouts where compatible.
- Convert static metadata.
- Map dynamic route segments.
- Generate `.contract.ts` stubs for ambiguous semantics.
- Emit migration report with skipped files and manual review items.

### Bench

```bash
needle bench hello-static
needle bench api-hot
needle bench large-build
needle bench --json
```

Benchmark commands are deferred until runtime paths and fixtures exist. Performance claims must not be published without reproducible fixtures.

## Prompting and CI Behavior

Interactive prompts are allowed only in local human mode.

Commands must not prompt when:

- `--json` is set.
- `--ci` is set.
- The command is called by MCP.
- The command is called by an agent plan or safe edit transaction.

In those cases, return a diagnostic explaining what approval or flag is required.

## Diagnostics

CLI diagnostics should use stable codes.

Examples:

| Code | Meaning |
| --- | --- |
| `NS_CONFIG_NOT_FOUND` | Config file could not be found when required. |
| `NS_CONFIG_INVALID` | Config file loaded but failed validation. |
| `NS_ROUTE_COLLISION` | Two route files resolve to the same URL. |
| `NS_ROUTE_INVALID_SEGMENT` | Route segment syntax is invalid. |
| `NS_RENDER_INVALID_EXPORT` | `render` export is malformed. |
| `NS_SEO_MISSING_TITLE` | Public route is missing title metadata. |
| `NS_SEO_MISSING_CANONICAL` | Public route is missing canonical metadata. |
| `NS_MAP_LOW_CONFIDENCE` | Requested graph answer relies on low-confidence edges. |
| `NS_EDIT_REJECTED_FIELD` | Safe edit requested a non-editable field. |
| `NS_ADAPTER_UNSUPPORTED` | Selected adapter cannot support the route output. |

Diagnostics should include `file`, `route`, or `node` when available.

## Documentation Rule

When a command becomes implemented, update:

- `docs/status.md`
- this file
- `README.md` if the command is user-facing
- `AGENTS.md` if agents need the command
- package README for the owning package once package READMEs exist
